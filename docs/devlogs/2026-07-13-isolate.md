# Isolate Codex SQLite State Per Pane

Date: 2026-07-13
Release target: unreleased

## Summary

- Prevent concurrent Codex panes from failing at startup with SQLite `database is locked` errors.
- Tracks [#193](https://github.com/jasonsuhari/gridbash/issues/193).

## What Changed

- Lease every pane a unique persistent `CODEX_SQLITE_HOME`, including shell panes that launch Codex manually.
- Partition lanes by the effective `CODEX_HOME`, protect them with cross-process file locks, and reuse released lanes across GridBash launches.
- Allocate a fresh lane before replacing a restarted pane, keeping the old lane leased until the old pane is dropped.
- Preserve the shared `CODEX_HOME` used for auth, config, skills, and rollout history.
- Preserve native non-UTF-8 Unix paths when deriving lane scopes and child environment values.
- Respect explicit user overrides and replace only GridBash-managed values in nested GridBash sessions.

## Why It Matters

- GridBash can run many Codex processes side by side without making them compete for one local runtime database.
- Users keep one login and one Codex configuration instead of maintaining a full `CODEX_HOME` copy for every pane.
- Persistent reuse avoids forcing Codex to rebuild a lane's rollout index on every GridBash launch. A newly allocated lane still performs Codex's normal first-use indexing, and SQLite-only goals, memories, and relationships remain lane-local.
